# ADR-034: Nostr Relay NIP Conformance — NIP-65, Kind-10002, NIP-19 TLV, NIP-11 Update

## Status: Proposed

## Date: 2026-05-06

## Context

Several NIP conformance gaps in the DreamLab relay and client reduce interoperability with the broader Nostr ecosystem. These gaps are individually minor but collectively prevent DreamLab from functioning as a first-class Nostr relay that external clients can use without workarounds.

The first gap is the absence of kind-10002 (NIP-65 relay list metadata). When a DreamLab user's pubkey is encountered by an external client (Damus, Amethyst, Coracle, etc.), the client queries known relays for the user's kind-10002 event to discover which relays the user reads from and writes to. Without a kind-10002 event, the external client has no way to route new events to relays that the DreamLab user actually monitors — events may be delivered to relays the user never checks, causing missed mentions and DMs from outside the DreamLab ecosystem. DreamLab users effectively become invisible to the broader Nostr social graph.

The second gap is the absence of NIP-19 TLV encoding in nostr-core. NIP-19 defines `nprofile`, `nevent`, `naddr`, `nrelay`, and `nsec`/`npub` bech32 encodings. The `nprofile` format encodes both a pubkey and one or more relay hints, allowing clients to share a user profile link that includes routing information. The `nevent` format similarly encodes an event ID with relay hints. Without NIP-19 support in nostr-core, the forum client cannot generate shareable profile or event links in standard format. Any `nostr:nprofile1...` link generated by another client and shared in the forum cannot be parsed. The `bech32 = "0.11"` crate is already in the workspace.

The third gap is the relay's NIP-11 document. The `supported_nips` array in `relay-worker/src/nip11.rs` does not include NIP-17, NIP-26, NIP-59, NIP-65, or NIP-90, even though these NIPs are either implemented or in progress via current ADRs. External clients that check `supported_nips` before connecting will incorrectly assume these protocols are not supported and may not send the corresponding event kinds to the relay. The NIP-11 document is also missing the `icon`, `banner`, `retention`, and `fees` fields that modern clients display.

The fourth gap is the relay's NIP-42 AUTH challenge behaviour. Per the NIP-42 specification, section 2: "When the relay wants to authenticate the client, it must send an AUTH message." The current relay implementation only validates AUTH events — it never proactively sends the `["AUTH", "<challenge>"]` message on connection establishment. Clients that check for an AUTH challenge on connect (Coracle, Nostr-tools WebSocket relay) may not send AUTH at all, causing authenticated-only routes (kind-1059 delivery, admin routes) to be less reliable. Sending the AUTH challenge on connect is fully backward-compatible — clients that do not implement NIP-42 will ignore it.

## Decision

Implement kind-10002 auto-publishing, add NIP-19 TLV encoding/decoding to nostr-core, update the NIP-11 document, and add proactive AUTH challenge on relay WebSocket connection.

### 1. Kind-10002 Auto-Publish on First Login

In `forum-client/src/stores/auth.rs`, after successful authentication (passkey PRF, NIP-07, or NIP-46), check for existing kind-10002 and publish one if absent:

```rust
// forum-client/src/auth/post_auth.rs

pub async fn ensure_relay_list_published(
    state: &AuthState,
    relay_pool: &RelayPool,
) -> Result<(), PostAuthError> {
    // Check if this pubkey has a kind-10002 on the relay
    let existing = relay_pool.query_first(NostrFilter {
        authors: Some(vec![state.pubkey_hex()]),
        kinds: Some(vec![10002]),
        limit: Some(1),
    }).await;

    if existing.is_some() {
        return Ok(()); // Already published
    }

    // Build kind-10002 relay list event
    // NIP-65: tags are ["r", relay_url] or ["r", relay_url, "read"|"write"]
    // No marker = both read and write
    let tags = vec![
        vec!["r".to_string(), "wss://relay.dreamlab-ai.com".to_string()],
    ];

    let unsigned = UnsignedEvent {
        pubkey: state.public_key(),
        kind: 10002,
        content: String::new(),
        tags,
        created_at: current_unix_timestamp(),
    };

    let signed = state.sign_event(unsigned).await
        .map_err(PostAuthError::Signer)?;

    relay_pool.publish(signed).await
        .map_err(PostAuthError::Publish)?;

    Ok(())
}
```

This function is called in the authentication completion handler, after the whitelist check passes. It runs at most once per user per relay (idempotent due to the existence check). Kind-10002 is in the replaceable range (10000-19999) — the relay already handles it correctly as a replaceable event.

### 2. NIP-19 TLV Encoding/Decoding

Add `nostr-core/src/nip19.rs`. All functions use `bech32 = "0.11"` (already in workspace `Cargo.toml`):

```rust
//! NIP-19: Bech32-encoded entities with optional TLV metadata.

use bech32::{Bech32, Hrp};

const HRP_NPUB: &str = "npub";
const HRP_NSEC: &str = "nsec";
const HRP_NOTE: &str = "note";
const HRP_NPROFILE: &str = "nprofile";
const HRP_NEVENT: &str = "nevent";
const HRP_NADDR: &str = "naddr";
const HRP_NRELAY: &str = "nrelay";

// TLV type bytes
const TLV_SPECIAL: u8 = 0;  // pubkey / event_id / identifier
const TLV_RELAY: u8 = 1;
const TLV_AUTHOR: u8 = 2;
const TLV_KIND: u8 = 3;

#[derive(Debug, Clone)]
pub struct Nprofile {
    pub pubkey: [u8; 32],
    pub relays: Vec<String>,
}

#[derive(Debug, Clone)]
pub struct Nevent {
    pub event_id: [u8; 32],
    pub relays: Vec<String>,
    pub author: Option<[u8; 32]>,
    pub kind: Option<u32>,
}

#[derive(Debug, Clone)]
pub struct Naddr {
    pub identifier: String,
    pub pubkey: [u8; 32],
    pub kind: u32,
    pub relays: Vec<String>,
}

/// Encode an npub (bare pubkey, no relay hints).
pub fn encode_npub(pubkey: &[u8; 32]) -> Result<String, Nip19Error> {
    bech32::encode::<Bech32>(
        Hrp::parse(HRP_NPUB).map_err(|e| Nip19Error::Encode(e.to_string()))?,
        pubkey,
    ).map_err(|e| Nip19Error::Encode(e.to_string()))
}

/// Decode an npub string to raw 32-byte pubkey.
pub fn decode_npub(npub: &str) -> Result<[u8; 32], Nip19Error> {
    let (hrp, data) = bech32::decode(npub)
        .map_err(|e| Nip19Error::Decode(e.to_string()))?;
    if hrp.as_str() != HRP_NPUB {
        return Err(Nip19Error::WrongHrp { expected: HRP_NPUB.into(), got: hrp.to_string() });
    }
    data.try_into().map_err(|_| Nip19Error::WrongLength { expected: 32, got: data.len() })
}

/// Encode an nprofile (pubkey + relay hints).
pub fn encode_nprofile(profile: &Nprofile) -> Result<String, Nip19Error> {
    let mut bytes = Vec::new();
    // TLV type 0: special (pubkey)
    bytes.push(TLV_SPECIAL);
    bytes.push(32u8);
    bytes.extend_from_slice(&profile.pubkey);
    // TLV type 1: relay (one entry per relay)
    for relay in &profile.relays {
        let relay_bytes = relay.as_bytes();
        bytes.push(TLV_RELAY);
        bytes.push(relay_bytes.len() as u8);
        bytes.extend_from_slice(relay_bytes);
    }
    bech32::encode::<Bech32>(
        Hrp::parse(HRP_NPROFILE).map_err(|e| Nip19Error::Encode(e.to_string()))?,
        &bytes,
    ).map_err(|e| Nip19Error::Encode(e.to_string()))
}

/// Decode an nprofile string.
pub fn decode_nprofile(nprofile: &str) -> Result<Nprofile, Nip19Error> {
    let (hrp, data) = bech32::decode(nprofile)
        .map_err(|e| Nip19Error::Decode(e.to_string()))?;
    if hrp.as_str() != HRP_NPROFILE {
        return Err(Nip19Error::WrongHrp { expected: HRP_NPROFILE.into(), got: hrp.to_string() });
    }
    parse_nprofile_tlv(&data)
}

fn parse_nprofile_tlv(data: &[u8]) -> Result<Nprofile, Nip19Error> {
    let mut pubkey: Option<[u8; 32]> = None;
    let mut relays = Vec::new();
    let mut pos = 0;
    while pos < data.len() {
        let tlv_type = data[pos]; pos += 1;
        let tlv_len = *data.get(pos).ok_or(Nip19Error::TruncatedTlv)? as usize; pos += 1;
        let value = data.get(pos..pos + tlv_len).ok_or(Nip19Error::TruncatedTlv)?;
        pos += tlv_len;
        match tlv_type {
            TLV_SPECIAL => {
                pubkey = Some(value.try_into()
                    .map_err(|_| Nip19Error::WrongLength { expected: 32, got: value.len() })?);
            },
            TLV_RELAY => {
                relays.push(String::from_utf8(value.to_vec())
                    .map_err(|e| Nip19Error::Utf8(e.to_string()))?);
            },
            _ => { /* Unknown TLV type — ignore per NIP-19 spec */ }
        }
    }
    Ok(Nprofile {
        pubkey: pubkey.ok_or(Nip19Error::MissingSpecial)?,
        relays,
    })
}

/// Encode an nevent (event ID + optional relay hints + optional author + optional kind).
pub fn encode_nevent(event: &Nevent) -> Result<String, Nip19Error> {
    let mut bytes = Vec::new();
    bytes.push(TLV_SPECIAL); bytes.push(32u8);
    bytes.extend_from_slice(&event.event_id);
    for relay in &event.relays {
        let rb = relay.as_bytes();
        bytes.push(TLV_RELAY); bytes.push(rb.len() as u8); bytes.extend_from_slice(rb);
    }
    if let Some(ref author) = event.author {
        bytes.push(TLV_AUTHOR); bytes.push(32u8); bytes.extend_from_slice(author);
    }
    if let Some(kind) = event.kind {
        bytes.push(TLV_KIND); bytes.push(4u8);
        bytes.extend_from_slice(&kind.to_be_bytes());
    }
    bech32::encode::<Bech32>(
        Hrp::parse(HRP_NEVENT).map_err(|e| Nip19Error::Encode(e.to_string()))?,
        &bytes,
    ).map_err(|e| Nip19Error::Encode(e.to_string()))
}

/// Encode an naddr (parameterised replaceable event address).
pub fn encode_naddr(addr: &Naddr) -> Result<String, Nip19Error> {
    let mut bytes = Vec::new();
    let id_bytes = addr.identifier.as_bytes();
    bytes.push(TLV_SPECIAL); bytes.push(id_bytes.len() as u8); bytes.extend_from_slice(id_bytes);
    for relay in &addr.relays {
        let rb = relay.as_bytes();
        bytes.push(TLV_RELAY); bytes.push(rb.len() as u8); bytes.extend_from_slice(rb);
    }
    bytes.push(TLV_AUTHOR); bytes.push(32u8); bytes.extend_from_slice(&addr.pubkey);
    bytes.push(TLV_KIND); bytes.push(4u8); bytes.extend_from_slice(&addr.kind.to_be_bytes());
    bech32::encode::<Bech32>(
        Hrp::parse(HRP_NADDR).map_err(|e| Nip19Error::Encode(e.to_string()))?,
        &bytes,
    ).map_err(|e| Nip19Error::Encode(e.to_string()))
}

#[derive(Debug, thiserror::Error)]
pub enum Nip19Error {
    #[error("bech32 encode error: {0}")]
    Encode(String),
    #[error("bech32 decode error: {0}")]
    Decode(String),
    #[error("wrong HRP: expected {expected}, got {got}")]
    WrongHrp { expected: String, got: String },
    #[error("wrong data length: expected {expected} bytes, got {got}")]
    WrongLength { expected: usize, got: usize },
    #[error("truncated TLV data")]
    TruncatedTlv,
    #[error("missing required TLV type 0 (special)")]
    MissingSpecial,
    #[error("UTF-8 decode error: {0}")]
    Utf8(String),
}

#[cfg(test)]
mod tests {

    }

    #[test]

}
```

### 4. Proactive AUTH Challenge on Connect

In `relay-worker/src/relay_do/mod.rs`, in the WebSocket connection handler, immediately after accepting the WebSocket upgrade:

```rust
// relay_do/mod.rs — WebSocket handler
pub async fn handle_websocket(
    &mut self,
    ws: WebSocket,
    server: WebSocketPair,
) -> Result<(), RelayError> {
    let session = RelaySession::new();

    // NIP-42: Send AUTH challenge immediately on connect
    // This makes the relay fully conformant with NIP-42 §2
    let challenge = generate_auth_challenge();  // UUID v4
    let auth_msg = serde_json::json!(["AUTH", challenge]);
    server.send_with_str(&serde_json::to_string(&auth_msg)?)?;

    // Store challenge in session for later verification
    let mut session = session;
    session.pending_auth_challenge = Some(challenge);

    // Continue with normal WebSocket message loop
    self.handle_messages(ws, server, session).await
}

fn generate_auth_challenge() -> String {
    // Generate a random challenge string (UUID v4 format)
    // In wasm32 context, use getrandom::getrandom for entropy
    let mut bytes = [0u8; 16];
    getrandom::getrandom(&mut bytes).expect("getrandom available in CF Workers");
    // Format as UUID v4
    format!(
        "{:08x}-{:04x}-4{:03x}-{:04x}-{:012x}",
        u32::from_be_bytes(bytes[0..4].try_into().unwrap()),
        u16::from_be_bytes(bytes[4..6].try_into().unwrap()),
        u16::from_be_bytes(bytes[6..8].try_into().unwrap()) & 0x0fff,
        (u16::from_be_bytes(bytes[8..10].try_into().unwrap()) & 0x3fff) | 0x8000,
        {
            let high = u32::from_be_bytes(bytes[10..14].try_into().unwrap()) as u64;
            let low = u16::from_be_bytes(bytes[14..16].try_into().unwrap()) as u64;
            (high << 16) | low
        }
    )
}
```

The `RelaySession` struct gains a `pending_auth_challenge: Option<String>` field. The existing `handle_auth_event()` handler already reads a challenge from the session — it should use `pending_auth_challenge` rather than a globally stored value. The challenge is consumed (set to `None`) after first use, preventing replay.

## Consequences

### Positive

- Kind-10002 publishing makes DreamLab users discoverable to the broader Nostr ecosystem. External clients routing mentions and DMs can find `wss://relay.dreamlab-ai.com` as the preferred relay.
- NIP-19 TLV encoding/decoding enables shareable `nostr:nprofile1...` and `nostr:nevent1...` links in the forum UI. These links are standard across all Nostr clients.
- The updated NIP-11 `supported_nips` array accurately reflects the relay's capabilities. Clients that gate behaviour on supported NIPs (e.g., only sending NIP-26 delegation events to relays that list NIP-26) now work correctly with DreamLab.
- Proactive AUTH challenge sends a clear signal to connecting clients that this relay requires authentication. Clients that support NIP-42 will authenticate immediately, avoiding a round-trip delay.
- `auth_required: true` in `RelayLimitation` tells clients upfront that unauthenticated connections are limited, reducing unnecessary connection attempts from public crawlers.

### Negative / Trade-offs

- Proactive AUTH challenge adds one additional WebSocket message to every connection sequence. For clients that do not use authenticated routes (read-only external clients), this is an unnecessary message. However, the message is small (under 100 bytes) and backward-compatible.
- Kind-10002 auto-publication on login adds a relay query + potential publish event to the authentication flow. On slow connections, this extends the time to first-interactive by up to 500ms. Mitigation: run the kind-10002 check in a background `spawn_local` task after the auth state is set, not as a blocking step in the auth flow.
- The NIP-11 `retention` and `limitation` fields are static in the `Default` implementation but should ideally be read from the D1 `settings` table (as defined in ADR-026). The static default is acceptable for the initial implementation; make these dynamic in Phase 2.
- `decode_nevent` is not implemented in this ADR (marked TODO). This means `nostr:nevent1...` URIs in message content are not parsed. Implement as a follow-up to avoid delaying the other changes.

### Neutral

- The `bech32 = "0.11"` crate is already in the workspace — no new dependency added for NIP-19.
- The NIP-11 update is a JSON change to a static struct — zero runtime cost change.
- Kind-10002 is already stored as a replaceable event by the relay (kinds 10000-19999 are already replaceable) — no relay storage change needed.

## Options Considered

### Option 1: Implement NIP-19 using a third-party Nostr crate (nostr = "0.44.2")
- **Pros**: Less code to write; battle-tested implementation.
- **Cons**: `nostr = "0.44.2"` already in the workspace but its NIP-19 module may have API differences from what the forum client needs. Keeping NIP-19 in nostr-core keeps the API consistent with the rest of the forum's Nostr types.

### Option 2: Defer kind-10002 and NIP-19 to when they are explicitly needed
- **Pros**: Reduced scope.
- **Cons**: DreamLab users are already invisible to the external Nostr ecosystem because of these gaps. Every day without kind-10002 is a day of missed interactions for users who have external Nostr contacts.

### Option 3: Use NIP-19 only for display (encode only, no decode)
- **Pros**: Simpler; no TLV parsing required.
- **Cons**: DreamLab forum content (messages containing `nostr:nprofile1...` from external shares) would display as raw strings rather than rich profile links. This degrades the experience for users who share Nostr URIs in chat.

### Option 4: Proactive AUTH only for specific routes (not on all connections)
- **Pros**: Reduces noise for read-only connections.
- **Cons**: NIP-42 spec says the relay "must send" AUTH when it wants to authenticate. Selective challenge breaks the spec's intent. The per-connection overhead is negligible.

- ADR-023: Forum relay hardening — NIP-42 session management that the proactive AUTH challenge integrates with.
- ADR-031: DM protocol — kind-10050 (preferred DM relay) is analogous to kind-10002 and benefits from the same auto-publish pattern.
- ADR-030: Authentication signer abstraction — kind-10002 publishing uses the `Signer` trait.
