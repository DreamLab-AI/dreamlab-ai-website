# Security Overview — DreamLab Community Forum (Rust Port)

## Compile-Time Safety

All crates enforce `#![deny(unsafe_code)]` at the crate root. The Rust ownership
model eliminates use-after-free, double-free, and data races at compile time. No
`unsafe` blocks are permitted; any required FFI boundary (e.g., `wasm-bindgen`)
uses safe wrapper types generated by the bindgen toolchain.

## Cryptographic Stack

| Operation | Crate | Audit Status |
|-----------|-------|--------------|
| secp256k1 / BIP340 Schnorr | `k256` 0.13.x | NCC Group audited |
| NIP-44 AEAD cipher | `chacha20poly1305` 0.10.x | NCC Group audited (RustCrypto) |
| HKDF key derivation | `hkdf` 0.12.x | RustCrypto, pure Rust |
| SHA-256 hashing | `sha2` 0.10.x | RustCrypto, pure Rust |
| CSPRNG for WASM | `getrandom` 0.2.x | **Must** enable `js` feature flag |

The `getrandom` crate requires `features = ["js"]` in every crate's `Cargo.toml`
that targets `wasm32-unknown-unknown`. Without this feature, any call to
`getrandom::getrandom()` panics at runtime in the browser or Workers environment.
This is enforced by a CI check that greps all workspace `Cargo.toml` files.

## WebAuthn PRF Key Derivation

Passkey authentication uses `passkey-rs` (maintained by 1Password) with PRF
extension support. The PRF output (32 bytes from the authenticator's HMAC-SHA-256)
is fed through HKDF to derive a deterministic secp256k1 private key. The private
key is never stored — it exists only in a Rust closure (`Option<SecretKey>`) and
is zeroized on page unload via the `zeroize` crate.

Key constraints:
- Cross-device QR authentication is blocked (produces different PRF output)
- Windows Hello is blocked (no PRF extension support)
- USB security keys are allowed (consistent PRF output)

## NIP-98 HTTP Authentication

All state-mutating API calls require a `Nostr` authorization header containing a
base64-encoded, Schnorr-signed kind 27235 event. The `nostr-core` crate provides
both client-side signing and server-side verification. Verification recomputes the
event ID from NIP-01 canonical serialization and checks the Schnorr signature
independently of the `id` field to prevent ID spoofing. Payload hashes (SHA-256 of
raw request body) are mandatory when a body is present.

## Zone-Based Access Control

The forum implements 4 access zones with cohort-based hierarchy:

| Zone | Visibility | Write Access |
|------|-----------|-------------|
| Public Lobby | All whitelisted users | All whitelisted users |
| Cohort Channels | Members of the cohort | Members of the cohort |
| Staff Lounge | Staff + Admin | Staff + Admin |
| Admin Zone | Admin only | Admin only |

Zone enforcement happens at the relay level (whitelist check before event storage)
and at the client level (UI filtering based on user cohorts from the auth store).

## CORS Configuration

Allowed origins for all Workers:

- `https://dreamlab-ai.com` (production)
- `https://thedreamlab.uk` (alternate domain)
- `https://dreamlab-ai.github.io` (GitHub Pages)
- `http://localhost:5173` (Vite dev server)
- `http://localhost:5174` (alternate dev port)

Origins are validated per-request. Wildcard origins are never used.

## Input Validation

All Worker request boundaries use `serde` for deserialization (compile-time schema
enforcement) and `validator` for runtime constraint checking (string length, regex
patterns, numeric ranges). Invalid input is rejected with structured JSON error
responses before any business logic executes.

Specific limits enforced at the relay:
- Max content size: 64KB (8KB for registration events)
- Max tag count: 2,000
- Max tag value size: 1,024 bytes
- Max timestamp drift: 7 days
- Rate limit: 10 events/second per IP
- Max connections: 20 per IP
- Max subscriptions: 20 per WebSocket

## SSRF Protection

The link-preview Worker validates all target URLs against a blocklist of private,
loopback, link-local, and cloud metadata IP ranges. Integer and hex IP obfuscation
formats are explicitly blocked. Only `http:` and `https:` protocols are allowed.

## Dependency Policy

- All crate versions are pinned in `Cargo.lock` (committed to repository)
- `nostr-sdk` 0.44.x is alpha — pinned to exact patch versions
- `cargo audit` runs in CI on every push
- `cargo clippy -- -D warnings` must pass with zero warnings
