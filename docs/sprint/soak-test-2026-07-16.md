# Forum soak test — 10-persona browser pass (2026-07-16)

A structured browser soak of the upstream `nostr-bbs-forum-client` (the
`/community/` SPA), driven through the browser sidecar (Chrome Beta 149,
NVIDIA Vulkan) against a locally-served Trunk build. The goal was to exercise
the full first-run journey across a spread of user tiers and adversarial display
names, then fix every defect **upstream in the kit** rather than papering over it
in the overlay.

All 16 issues are resolved in the kit working tree; the changes land in the kit
`CHANGELOG.md` "Unreleased" section and are confined to `nostr-bbs-forum-client`.
Photo-upload end-to-end remains **pending** on a local pod-worker (see caveats).

Severity legend (kit house style): **SEV1** broken/blocking · **SEV2** wrong
behaviour · **SEV3** polish.

---

## Method

### Harness — the `dev-auth` feature

To soak the UI without a live relay whitelist round-trip, the kit gained a
`dev-auth` cargo feature (off by default, never in release builds). It provides:

- three pre-provisioned **deterministic** identities (`Dev Admin`, `Dev User`,
  `Junkie Jarvis`) built from fixed 32-byte seeds — not secret, dev-only;
- a floating identity-picker panel (`DevAuthPanel`) that logs in instantly;
- a **zone-access bypass** (`dev_apply_zone_access`) that applies each identity's
  cohorts/admin flag locally instead of calling `/api/check-whitelist`;
- a local **jarvis DM echo-bot** (`spawn_jarvis_echo_bot`) that watches outgoing
  DMs to the jarvis pubkey and injects a canned reply into the DM store.

The seven remaining personas were created through the live **signup wizard** so
the brand-new-user path (kind-0 publication, auto-whitelist, onboarding capture)
was itself under test.

### Resolution detail

- **#1–#3 (polish).** Breadcrumb `/` → `.breadcrumb-separator` `›` on Settings,
  Events, Glossary, Note and Pod-browser; nav label "Agents" → "Governance"
  (desktop + mobile); welcome greeting trims trailing whitespace/ASCII
  punctuation before appending its own.
- **#4 (governance).** `GovernancePage` now branches on `member_view`: members
  get read-only "About governance" copy; admins get the management-surface copy
  naming Approve/Reject and a "Register an agent →" link.
- **#5 (whitelist).** `app.rs` publishes a **minimal** kind-0 for a user with no
  display name and no claimed handle **only when the relay holds none**, so
  auto-whitelist can proceed without clobbering an existing profile.
- **#6 (settings).** The Username card is now reactive on `claimed_username` and
  renders only when a legacy handle still resolves (release-only). Onboarding no
  longer mints handles, so a fresh account simply has no Username section.
- **#7 (empty zone).** `CategoryPage` shows admins a "Create a section" CTA
  deep-linking to `/admin?tab=channels` (sections *are* channels; the "Sections"
  tab is join-request approvals, not creation). Non-admins keep the info copy.
- **#8/#10/#12/#15 (notifications).** Rewrite of the producer sync state:
  baseline + dedup are keyed **per pubkey** (`nostrbbs:notif_sync:<pubkey>`), so
  a recovery-key login on a shared device starts fresh. A one-time backlog
  snapshot records already-known members without notifying; own-join is
  suppressed; a member with any pre-baseline post is treated as existing; genuine
  post-baseline joins (Carol Chen, #15) still notify. The old
  mark-known-before-checks ordering that permanently suppressed a late join is
  gone.
- **#9 (drawer).** `NotificationCenter` replaces the full-viewport backdrop
  `<div>` with a capture-phase document `pointerdown` listener that closes the
  drawer **without consuming the event**, so the control behind it (e.g. Log Out)
  still receives the click. Bell and panel are excluded via `closest()` markers.
- **#11 (recovery).** `AuthStore::hydrate_profile_from_relay` fetches the
  account's own kind-0 after an nsec/local-key login and hydrates
  nickname/avatar + the persisted session. Fire-and-forget, silent on failure,
  and only applies a field untouched since login.
- **#13 (length).** `MAX_NICKNAME_LEN` (50) drives the input `maxlength`, a live
  `n/50` counter, and honest helper/at-limit copy — one const so they cannot
  drift.
- **#14/#16 (errors + upload).** `pod_client`, `zone_access` and the channel
  create path now emit friendly messages and log the opaque `JsValue`/HTTP detail
  to the console. The avatar `<input>` is cleared only after the blob is safely
  read (or in blob-free early returns), fixing the `NotFoundError`.

## Environment caveats

- **Photo upload E2E is pending.** The `pod-worker` was not running locally, so
  avatar and media uploads could not be driven end-to-end against a real pod
  backend. The client-side fixes (#14/#16 — input-reset ordering, friendly
  errors) are verified in isolation; a full round-trip needs a live pod-worker.
- **Avatar `NotFoundError` may be partly a sidecar artifact.** The
  `NotFoundError` on blob read reproduced through the browser sidecar, whose file
  handling can differ from a user's local filesystem. The input-reset reorder
  (#16) is the correct fix regardless, but the exact trigger could be a
  sidecar file-path quirk; re-confirm on a real desktop browser once the
  pod-worker is up.
- **In-post image upload untested.** Forum channels/sections were empty during
  the soak, so there was no thread in which to attach an in-post image. That path
  (composer image attach) was therefore not exercised and should be revisited
  once seeded content exists.

## Follow-on — zone-first navigation redesign (ADR-107, upstream)

The soak surfaced that single-zone communities are confused by the generic
global forums index, and the operator asked for members to land in their own
zone. The response is a **zone-first landing and scoped navigation** redesign,
implemented **upstream in the kit** and recorded in the kit's
`docs/adr/ADR-107-zone-first-landing-and-scoped-navigation.md`. In brief: the
client derives a member's "home zone" from `ZONE_CONFIG × cohorts` via
`Zone::is_member` (`home_zone_for`) — the single accessible **locked** zone, with
admins exempt; the `/forums` index auto-forwards to that zone's root with replace
navigation; the header + mobile nav anchor and the zone-page breadcrumb re-root
on the zone; multi-zone members, admins, deep links and the `returnTo` flow
(ADR-090) are unchanged. See the ADR for the full behaviour matrix.

### Browser smoke result (2026-07-16)

Driven through the browser sidecar against the local Trunk dev server
(`http://agentbox:8090/`). Checks 3–5 required reproducing the production-intent
DreamLab `ZONE_CONFIG` via a pre-page `initScript`, because the live dev
`index.html` injects an empty `window.__ENV__` and therefore serves the **legacy
fallback** zones — under which the Dev Normal User's cohorts (`[home, friends]`)
intersect neither legacy locked zone (`members`/`private`), leaving the feature
correctly **dormant**.
